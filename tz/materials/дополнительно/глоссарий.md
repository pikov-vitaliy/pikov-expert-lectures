## 📚 Общие термины

| Термин | Определение | Контекст | Пример |
|-------|-------------|---------|--------|
| **Техническое задание (ТЗ)** | Документ, устанавливающий комплекс требований к объекту разработки, необходимых и достаточных для его разработки, а также проведения приёмочных испытаний | Управление проектами | ГОСТ 19.201-78 |
| **Требование** | Условие или возможность, которой должна соответствовать система или ее компонент | Инженерия требований | "Система должна предоставлять API" |
| **Заинтересованное лицо (Stakeholder)** | Лицо или организация, имеющая интерес к проекту или влияющая на него | Управление проектами | Заказчик, пользователь, разработчик |
| **Аналитик** | Специалист, занимающийся сбором, анализом и формализацией требований | Бизнес-анализ | Системный аналитик, бизнес-аналитик |
| **Архитектор** | Специалист, определяющий техническую архитектуру системы | Системная архитектура | Решение по выбору технологий |

## 🎯 Типы требований

| Термин | Определение | Категория | Пример |
|-------|-------------|----------|--------|
| **Функциональное требование (ФТ)** | Требование, определяющее, ЧТО должна делать система | Функциональные | "Система должна предоставлять форму входа" |
| **Нефункциональное требование (НФТ)** | Требование, определяющее, КАК система должна работать | Нефункциональные | "Время отклика API ≤ 200мс" |
| **Техническое требование** | Требование к технологиям, инструментам или окружению | Технические | "Использовать PostgreSQL" |
| **Бизнес-требование** | Требование, определяющее цели и задачи бизнеса | Бизнес | "Увеличить продажи на 20%" |
| **Пользовательское требование** | Требование, сформулированное с точки зрения пользователя | Пользовательские | "Пользователь должен иметь возможность restore пароль" |

## 📊 Нефункциональные требования

### Производительность

| Термин | Определение | Единица измерения | Пример |
|-------|-------------|-------------------|--------|
| **Время отклика** | Время, за которое система отвечает на запрос | мс, с | "≤ 200мс" |
| **Пропускная способность** | Количество запросов, которые система может обработать за единицу времени | запросов/с | "10,000 RPS" |
| **Загрузка системы** | Уровень использования ресурсов системы | % | "CPU < 70%" |
| **Время загрузки страницы** | Время, за которое страница полностью загружается в браузере | с | "≤ 2с" |
| **First Contentful Paint (FCP)** | Время от начала загрузки до отображения первого контента | с | "≤ 1с" |
| **Time to Interactive (TTI)** | Время, за которое страница становится интерактивной | с | "≤ 3с" |

### Надежность

| Термин | Определение | Единица измерения | Пример |
|-------|-------------|-------------------|--------|
| **Доступность (Availability)** | Процент времени, в течение которого система доступна для использования | % | "99.9%" |
| **Время наработки на отказ (MTBF)** | Среднее время между сбоями системы | часы, дни | "≥ 72 часа" |
| **Среднее время восстановления (MTTR)** | Среднее время, необходимое для восстановления системы после сбоя | минуты, часы | "≤ 30 минут" |
| **Отказоустойчивость** | Способность системы продолжать работу при отказе отдельных компонентов | - | "Да/Нет" |
| **Резервное копирование** | Процесс создания копий данных для восстановления | - | "Ежедневно в 02:00" |

### Удобство использования

| Термин | Определение | Стандарт | Пример |
|-------|-------------|----------|--------|
| **User Experience (UX)** | Впечатления и восприятие пользователя при взаимодействии с системой | - | "Интуитивно понятный интерфейс" |
| **User Interface (UI)** | Визуальный интерфейс системы | - | "Адаптивный дизайн" |
| **WCAG** | Руководство по доступности веб-контента | WCAG 2.1 | "Уровень AA" |
| **Usability** | Удобство и простота использования системы | - | "Коэффициент завершения задачи ≥ 90%" |
| **Accessibility** | Доступность системы для людей с ограниченными возможностями | - | "Поддержка клавиатурной навигации" |

### Масштабируемость

| Термин | Определение | Тип | Пример |
|-------|-------------|-----|--------|
| **Горизонтальное масштабирование** | Добавление новых серверов для увеличения мощности | Масштабирование | "Добавление новых узлов в кластер" |
| **Вертикальное масштабирование** | Увеличение ресурсов существующих серверов | Масштабирование | "Увеличение RAM с 8GB до 16GB" |
| **Автоматическое масштабирование** | Автоматическое добавление/удаление ресурсов в зависимости от нагрузки | Масштабирование | "Auto-scaling в AWS" |

### Совместимость

| Термин | Определение | Контекст | Пример |
|-------|-------------|---------|--------|
| **Кроссбраузерность** | Способность системы работать в разных браузерах | Веб-разработка | "Chrome, Firefox, Safari, Edge" |
| **Кроссплатформенность** | Способность системы работать на разных платформах | Разработка ПО | "Windows, macOS, Linux" |
| **Responsive Design** | Адаптация интерфейса под разные размеры экранов | Веб-дизайн | "Мобильные, планшеты, десктопы" |

### Безопасность

| Термин | Определение | Стандарт | Пример |
|-------|-------------|----------|--------|
| **Аутентификация** | Процесс проверки подлинности пользователя | - | "Логин/пароль, 2FA" |
| **Авторизация** | Процесс определения прав доступа пользователя | - | "RBAC, ABAC" |
| **Шифрование** | Процесс преобразования данных в нечитаемый формат | - | "AES-256, TLS 1.3" |
| **OWASP Top 10** | Топ 10 уязвимостей веб-приложений | OWASP | "Broken Access Control, Injection" |
| **DDoS-атака** | Атака, направленная на отказ в обслуживании | Безопасность | "Защита через Cloudflare" |
| **Brute Force** | Атака методом подбора пароля | Безопасность | "Ограничение попыток входа" |
| **XSS (Cross-Site Scripting)** | Уязвимость, позволяющая выполнять произвольный JavaScript | Безопасность | "CSP (Content Security Policy)" |
| **CSRF (Cross-Site Request Forgery)** | Уязвимость, позволяющая выполнять действия от имени пользователя | Безопасность | "CSRF-токены" |
| **SQL Injection** | Уязвимость, позволяющая выполнять произвольные SQL-запросы | Безопасность | "Параметризованные запросы" |

## 🏗️ Методологии и подходы

| Термин | Определение | Контекст | Пример |
|-------|-------------|---------|--------|
| **Waterfall (Каскадная модель)** | Последовательный подход к разработке | Управление проектами | "Сбор требований → Проектирование → Разработка → Тестирование" |
| **Agile** | Гибкий подход к разработке с итерациями | Управление проектами | "Scrum, Kanban" |
| **Scrum** | Agile-фреймворк с фиксированными итерациями (спринтами) | Управление проектами | "Sprint Planning, Daily Scrum, Sprint Review" |
| **Kanban** | Метод управления задачами с визуализацией | Управление проектами | "To Do → In Progress → Review → Done" |
| **MVP (Minimum Viable Product)** | Минимально жизнеспособный продукт с основными функциями | Разработка ПО | "Первая версия продукта" |
| **User Story** | Краткое описание требования с точки зрения пользователя | Agile | "Как пользователь, я хочу войти в систему, чтобы получить доступ" |
| **Use Case** | Сценарий использования системы с акторами и потоками | Анализ требований | "Авторизация пользователя" |
| **INVEST** | Критерий качества User Stories (Independent, Negotiable, Valuable, Estimable, Small, Testable) | Agile | - |
| **SMART** | Критерий качества целей (Specific, Measurable, Achievable, Relevant, Time-bound) | Управление проектами | - |

## 💼 Роли и ответственности

| Термин | Определение | Ответственность | Пример |
|-------|-------------|----------------|--------|
| **Product Owner (PO)** | Владелец продукта, определяющий приоритеты | Формулировка требований, приоритизация | "Определение, что нужно делать" |
| **Scrum Master** | Фасилитатор процесса Scrum | Устранение препятствий, поддержка команды | "Организация встреч, помощь команде" |
| **Development Team** | Команда разработчиков | Реализация требований | "Backend, Frontend, QA" |
| **Business Analyst (BA)** | Бизнес-аналитик | Сбор и анализ требований | "Интервью с заказчиком" |
| **System Analyst** | Системный аналитик | Технический анализ требований | "Проектирование системы" |
| **Architect** | Архитектор | Определение архитектуры системы | "Выбор технологий" |
| **QA Engineer** | Инженер по обеспечению качества | Тестирование системы | "Написание тестов" |
| **DevOps Engineer** | Инженер по эксплуатации | Настройка инфраструктуры | "CI/CD, мониторинг" |

## 📊 Метрики и KPI

| Термин | Определение | Единица измерения | Пример |
|-------|-------------|-------------------|--------|
| **KPI (Key Performance Indicator)** | Ключевой показатель эффективности | - | "Количество пользователей" |
| **Velocity** | Количество Story Points, выполненных командой за спринт | SP/sprint | "40 SP/sprint" |
| **Burn-down Rate** | Скорость выполнения задач в спринте | SP/day | "4 SP/day" |
| **Defect Rate** | Процент задач с дефектами | % | "≤ 5%" |
| **Deployment Frequency** | Частота деплоев | deployments/time | "10 deployments/month" |
| **Mean Time to Repair (MTTR)** | Среднее время на исправление дефекта | часы, дни | "≤ 4 часа" |
| **NPS (Net Promoter Score)** | Показатель лояльности клиентов | -100 до +100 | "> 40" |
| **Retention Rate** | Процент пользователей, вернувшихся в продукт | % | "30-day retention: 40%" |
| **Conversion Rate** | Процент пользователей, выполнивших целевое действие | % | "10%" |

## 🔧 Технические термины

| Термин | Определение | Контекст | Пример |
|-------|-------------|---------|--------|
| **API (Application Programming Interface)** | Интерфейс для взаимодействия с системой | Разработка ПО | "REST API, GraphQL API" |
| **Backend** | Серверная часть системы | Разработка ПО | "Node.js, Python, Java" |
| **Frontend** | Клиентская часть системы | Разработка ПО | "React, Vue.js, Angular" |
| **Database** | Система управления базами данных | Хранение данных | "PostgreSQL, MongoDB, MySQL" |
| **Cache** | Кэш — временное хранилище данных для ускорения доступа | Производительность | "Redis, Memcached" |
| **Load Balancer** | Система распределения нагрузки между серверами | Масштабируемость | "NGINX, HAProxy" |
| **CDN (Content Delivery Network)** | Сеть доставки контента | Производительность | "Cloudflare, Akamai" |
| **Microservices** | Архитектурный стиль, при котором приложение разделяется на kleine сервисы | Архитектура | "User Service, Order Service" |
| **Monolith** | Архитектурный стиль, при котором все компоненты системы представляют собой единое целое | Архитектура | "Одно приложение со всеми функциями" |
| **Container** | Изолированная среда для выполнения приложений | DevOps | "Docker" |
| **Orchestration** | Управление контейнерами | DevOps | "Kubernetes, Docker Compose" |
| **CI/CD** | Непрерывная интеграция и доставка | DevOps | "GitHub Actions, GitLab CI" |

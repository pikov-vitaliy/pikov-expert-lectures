## 📚 Общие вопросы

### 1. 📌 Что такое техническое задание (ТЗ)?

**Ответ:** Техническое задание (ТЗ) — это документ, устанавливающий комплекс требований к объекту разработки, необходимых и достаточных для его разработки, а также проведения приёмочных испытаний.

**Ключевые характеристики:**
- ✅ **Полнота** — содержит всю необходимую информацию
- ✅ **Однизначность** — исключает двойственное толкование
- ✅ **Проверяемость** — можно проверить выполнение требований
- ✅ **Актуальность** — соответствует текущим потребностям бизнеса
- ✅ **Реалистичность** — выполнимо в рамках заданных ограничений

---

### 2. 🤔 Зачем нужно ТЗ?

**Ответ:** Техническое задание нужно для:

1. **Четкого понимания целей** — все участники проекта понимают, что нужно сделать
2. **Снижения рисков** — минимизация рисков недопонимания между заказчиком и исполнителем
3. **Контроля качества** — возможность проверки выполнения требований
4. **Правовой защиты** — документ служит юридической основой для проекта
5. **Эффективного управления** — упрощение планирования и контроля хода работ
6. **Снижения затрат** — уменьшение риска доработок и переделок

> **💡 Статистика:** Проекты с качественным ТЗ срываются в 3 раза реже, чем проекты без ТЗ или с плохим ТЗ.

---

### 3. 👥 Кто разрабатывает ТЗ?

**Ответ:** Разработкой ТЗ занимаются:

| Роль | Ответственность | Участие |
|------|----------------|---------|
| **Заказчик** | Формулировка бизнес-требований, определение целей | ✅ Обязательно |
| **Аналитик** | Сбор и формализация требований, создание документации | ✅ Обязательно |
| **Архитектор** | Техническая проработка, определение архитектуры | ✅ Рекомендуется |
| **Проектировщик** | Детализация требований, создание спецификаций | ✅ По необходимости |
| **Юрист** | Правовая экспертиза, проверка соответствия законодательству | ✅ Рекомендуется |
| **Руководитель проекта** | Координация процесса, контроль сроков | ✅ Рекомендуется |

> **💡 Совет:** Вовлекайте всех заинтересованных лиц (stakeholders) в процесс создания ТЗ.

## 📝 Вопросы по структуре и содержанию

### 4. 📄 Какую структуру должно иметь ТЗ?

**Ответ:** Стандартная структура ТЗ включает:

1. **Введение** (наименование, основание, заказчик, исполнитель, сроки)
2. **Назначение и цели** (для чего создается система, какие проблемы решает)
3. **Характеристика объекта** (описание объекта автоматизации)
4. **Требования**
   - Функциональные требования
   - Нефункциональные требования
   - Технические требования
5. **Требования к безопасности**
6. **Состав и содержимое работ** (этапы, документы)
7. **Порядок контроля и приемки** (виды испытаний, критерии)
8. **Приложения** (диаграммы, схемы, глоссарий)

> **📌 см. также:** [Шаблоны ТЗ](../шаблоны/) в папке materials

---

### 5. 🎯 Сколько страниц должно быть в ТЗ?

**Ответ:** Оптимальный размер ТЗ зависит от сложности проекта:

| Тип проекта | Рекомендуемый размер |
|-------------|---------------------|
| Малый проект | 5-15 страниц |
| Средний проект | 15-50 страниц |
| Крупный проект | 50-200+ страниц |

> **⚠️ Внимание:** Качество важнее количества. Лучше 10 страниц четких требований, чем 100 страниц расплывчатых формулировок.

---

### 6. ⚖️ Как определить уровень детализации ТЗ?

**Ответ:** Уровень детализации зависит от:

1. **Сложности проекта** — чем сложнее проект, тем детальнее ТЗ
2. **Опыта команды** — опытная команда может работать с менее детализированным ТЗ
3. **Методологии разработки** — Agile позволяет более гибкий подход
4. **Срока проекта** — длительные проекты требуют более детального ТЗ

**Рекомендации:**
- **Классический подход:** Полное ТЗ со всеми деталями
- **Agile подход:** Высокоуровневое ТЗ + детализация в процессе
- **Гибридный подход:** Полное ТЗ для критичных частей + Agile для остального

---

### 7. 📊 Как правильно формулировать требования?

**Ответ:** Хорошее требование должно быть:

1. **Конкретным** — четко сформулированным
2. **Однизначным** — исключающим двойственное толкование
3. **Проверяемым** — можно проверить выполнение
4. **Атомарным** — одно требование = одна функция
5. **Независимым от реализации** — описывает ЧТО, а не КАК

**Примеры:**

❌ **Плохо:**
```
Система должна быть удобной для пользователей
```

✅ **Хорошо:**
```
Система должна предоставлять интерфейс, соответствующий WCAG 2.1 (уровень AA)
```

❌ **Плохо:**
```
Система должна быть быстрой
```

✅ **Хорошо:**
```
Страницы системы должны загружаться не более 2 секунд при нагрузке 1000 одновременных пользователей
```

## 🔧 Вопросы по типам требований

### 8. 🎯 Что такое функциональные требования?

**Ответ:** Функциональные требования (ФТ) — это требования, определяющие, **ЧТО** должна делать система.

**Признаки хорошего ФТ:**
- ✅ Конкретность и однозначность
- ✅ Проверяемость (можно тестировать)
- ✅ Atomicity (атомарность)
- ✅ Независимость от реализации

**Форматы записи:**
1. **User Story:** "Как [роль], я хочу [действие], чтобы [цель]"
2. **Традиционный:** "Система должна [действие]"
3. **Use Case:** Сценарий использования с акторами и потоками

---

### 9. ⚙️ Что такое нефункциональные требования?

**Ответ:** Нефункциональные требования (НФТ) — это требования, определяющие, **КАК** система должна работать.

**Основные категории НФТ:**

| Категория | Описание | Примеры |
|-----------|----------|---------|
| **Производительность** | Скорость работы системы | Время отклика, пропускная способность |
| **Надежность** | Стабильность работы системы | Доступность, время восстановления |
| **Удобство использования** | User Experience | WCAG, адаптивный дизайн |
| **Масштабируемость** | Способность к росту | Горизонтальное/вертикальное масштабирование |
| **Безопасность** | Защита системы | Аутентификация, шифрование |
| **Совместимость** | Работа в разных средах | Браузеры, устройства, ОС |
| **Обслуживаемость** | Легкость поддержки | Логирование, мониторинг |

---

### 10. 🔒 Какие требования к безопасности должны быть в ТЗ?

**Ответ:** В ТЗ должны быть указаны:

1. **Общие требования:**
   - Соответствие ФЗ-152 "О персональных данных"
   - Соответствие ГОСТ Р ИСО/МЭК 27001-2022
   - Защита от OWASP Top 10 уязвимостей

2. **Аутентификация и авторизация:**
   - Методы аутентификации (логин/пароль, 2FA, биометрия)
   - Требования к паролям (длина, сложность, срок действия)
   - Модель разграничения доступа (RBAC, ABAC)

3. **Защита данных:**
   - Шифрование данных в покое (AES-256)
   - Шифрование данных в движении (TLS 1.3)
   - Хранение ключей шифрования

4. **Защита от угроз:**
   - Защита от DDoS-атаки
   - Защита от Brute Force
   - Защита от XSS, CSRF, SQL Injection

5. **Резервное копирование:**
   - Частота бэкапов
   - Хранение бэкапов
   - Время восстановления

---

### 11. 📈 Какие требования к производительности должны быть в ТЗ?

**Ответ:** В ТЗ должны быть указаны:

1. **Время отклика:**
   - API: ≤ 200 мс (p95)
   - Загрузка страницы: ≤ 2 с (First Contentful Paint)

2. **Пропускная способность:**
   - Максимальная нагрузка: [X] запросов/с
   - Одновременные пользователи: [X]

3. **Ресурсы:**
   - CPU: < 70% при пиковой нагрузке
   - RAM: < 80% при пиковой нагрузке
   - Disk I/O: < 80% при пиковой нагрузке

**Пример:**
```yaml
Производительность:
  Время отклика:
    - API: ≤ 200ms (p95)
    - Frontend: ≤ 1s (First Contentful Paint)
  Пропускная способность:
    - Максимальная нагрузка: 10,000 RPS
    - Одновременные пользователи: 5,000
  Ресурсы:
    - CPU: < 70%
    - RAM: < 80%
    - Disk I/O: < 80%
```

## 📅 Вопросы по процессу

### 12. ⏳ Сколько времени занимает создание ТЗ?

**Ответ:** Время создания ТЗ зависит от сложности проекта:

| Тип проекта | Срок создания ТЗ |
|-------------|-------------------|
| Малый проект | 1-3 дня |

| Очень крупный проект | 1-2 месяца |

### 13. 👥 Как вовлечь заинтересованных лиц в процесс создания ТЗ?

**Ответ:** Для эффективного вовлечения stakeholders:

1. **Определите всех заинтересованных лиц**
2. **Проведите интервью** с каждым из них
3. **Организуйте совместные сессии** (workshops)
4. **Показывайте промежуточные результаты**
5. **Собирайте обратную связь** на каждом этапе
6. **Проводите ревью документа** перед финализацией

**Методы сбора требований:**
- Интервью
- Анкетирование
- Наблюдение за работой пользователей
- Анализ существующих систем
- Brainstorming
- Прототипирование

---

### 14. ✅ Как проверить качество ТЗ перед передачей в разработку?

**Ответ:** Используйте чек-лист проверки:

1. **Полнота** — все необходимые разделы присутствуют
2. **Конкретность** — требования четки и однозначны
3. **Проверяемость** — каждое требование можно проверить
4. **Непротиворечивость** — требования не противоречат друг другу
5. **Реалистичность** — требования выполнимы в рамках ограничений
6. **Актуальность** — требования соответствуют текущим потребностям

> **📌 см. также:** [Чек-лист проверки ТЗ](чек-лист-ТЗ.md)

---

### 15. 📝 Как управлять изменениями в ТЗ?

**Ответ:** Для управления изменениями:

1. **Ведите журнал изменений** — фиксируйте все изменения
2. **Используйте систему контроля версий** (Git, SVN)
3. **Определите процесс утверждения изменений**
4. **Оценивайте влияние изменений** на сроки и бюджет
5. **Коммуницируйте изменения** всем заинтересованным лицам

**Пример журнала изменений:**

| Версия | Дата | Автор | Описание изменений |
|--------|------|-------|---------------------|
| 1.0 | 01.06.2026 | И.И. Иванов | Создание документа |
| 1.1 | 05.06.2026 | П.П. Петров | Добавлены требования к безопасности |
| 1.2 | 10.06.2026 | С.С. Сидоров | Изменены требования к производительности |

### 17. 📊 Как оценить стоимость реализации требований из ТЗ?

**Ответ:** Для оценки стоимости:

1. **Разбейте проект на задачи**
2. **Оцените каждую задачу** (в часах или Story Points)
3. **Определите ставку специалистов**
4. **Посчитайте итоговую стоимость**

**Методы оценки:**
- **Экспертная оценка** — оценка специалистами
- **Аналогия** — сравнение с похожими проектами
- **Parametric** — использование статистических данных
- **Bottom-up** — детализированная оценка каждой задачи

**Пример:**
```
Задача: Разработка модуля авторизации
Оценка: 40 часов
Ставка: 5,000 ₽/час
Стоимость: 40 * 5,000 = 200,000 ₽
```

## 🔍 Вопросы по специфическим ситуациям

### 20. 🤔 Нужно ли ТЗ для маленького проекта?

**Ответ:** Да, ТЗ нужно для любого проекта, но уровень детализации может отличаться.

**Для малых проектов:**
- Используйте упрощенный шаблон
- Фокусируйтесь на ключевых требованиях
- Избегайте избыточной детализации

**Пример упрощенного ТЗ:**
1. Введение
2. Цели и задачи
3. Функциональные требования
4. Нефункциональные требования
5. Сроки и этапы
6. Критерии приемки

> **📌 см. также:** [Шаблон для малых проектов](../шаблоны/шаблон-малый-проект.md)

---

### 21. 🚀 Как создать ТЗ для стартапа?

**Ответ:** Для стартапов рекомендуется:

1. **Начните с Product Vision** — определите цель и уникальную ценность продукта
2. **Используйте MVP подход** — создайте минимально жизнеспособный продукт
3. **Фокусируйтесь на ключевых функциях** — только то, без чего продукт не может существовать
4. **Будьте гибкими** — готовьтесь к изменениям
5. **Используйте Agile подход** — User Stories, Sprint Planning

**Рекомендуемый шаблон:** [Шаблон для MVP](../шаблоны/шаблон-MVP.md)

---

### 22. 🏢 Как создать ТЗ для госзаказа?

**Ответ:** Для госзаказов:

1. **Используйте стандартные шаблоны** (ГОСТ 19.201-78, ГОСТ 34.602-2020)
2. **Соблюдайте все формальные требования**
3. **Указывайте соответствие стандартам**
4. **Добавьте раздел по безопасности** (ФЗ-152, ГОСТ Р ИСО/МЭК 27001)
5. **Указывайте требования к документации**

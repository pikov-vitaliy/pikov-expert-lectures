# Чеклист аудита лекции pikov.expert

Дата фиксации: 20.06.2026  
Применение: каждая доменная папка и каждая карточка из `_PROJECT\lectures.json`.

## 1. Идентификация

- Папка и URL совпадают с `_PROJECT\lectures.json`.
- Название карточки, `title`, `h1` и фактическое содержание говорят об одной теме.
- Есть понятное назначение материала: лекция, курс, практикум, справочник или методика.
- Есть ожидаемый результат обучения: что слушатель сможет сделать после изучения.

## 2. Техническая страница

- `index.html` открывается локально.
- Есть корректный `<title>`.
- Есть один главный `h1` или явно понятный главный заголовок.
- Есть `meta description` без пустого маркетингового шума.
- Есть canonical URL на соответствующий поддомен или страницу.
- OpenGraph заполнен: `og:title`, `og:description`, `og:type`, `og:url`, при наличии изображения - `og:image`.
- Нет ссылок на внутренние служебные пути `_PROJECT`, `_НА_УДАЛЕНИЕ_2026-06-20`, старые сборочные папки.
- Локальные материалы, изображения, CSS/JS и ссылки доступны из текущей структуры.

## 3. Педагогика

- Структура идет от цели и контекста к понятиям, практике и проверке.
- Есть связка "зачем это нужно специалисту".
- Термины вводятся до использования или объясняются рядом.
- Есть практические задания, кейсы, вопросы для самопроверки или критерии оценки.
- Уровень сложности соответствует заявленной аудитории.
- Нет перегрузки слайдами/карточками без учебной логики.

## 4. Актуальность ИБ и РБПО

Проверять применимость, а не механически вставлять все стандарты в каждую лекцию.

- ГОСТ Р 56939-2024: процессы безопасной разработки, требования, проектирование, кодирование, испытания, сопровождение, управление уязвимостями.
- ФСТЭК: применимые приказы и методические документы по защите информации, КИИ, сертификации, оценке соответствия, безопасной разработке.
- ГОСТ Р 71206-2024 и ГОСТ Р 71207-2024: если лекция касается анализа требований, архитектуры, кода, тестирования или инструментального контроля.
- NIST SSDF SP 800-218: secure software development practices, evidence, verification.
- ISO/IEC 27001/27034: управление ИБ и application security.
- OWASP, CWE, MITRE ATT&CK: если есть уязвимости, модели атак, веб/API/приложения.
- CERT C/C++, MISRA C/C++: если материал касается C/C++, безопасного кодирования или SAST.

## 5. Secure C/C++ и SSDLC

Для лекций про разработку, SAST, ППК, РБПО, fuzzing и требования проверить наличие или уместность:

- memory safety: bounds checking, lifetime, ownership, integer overflow, use-after-free, undefined behavior;
- запрет/ограничение опасных API;
- компиляторные меры: `-Wall`, `-Wextra`, `-Werror` по политике проекта, `-fstack-protector-strong`, `-D_FORTIFY_SOURCE=2/3`, PIE/RELRO/NX, CFI при наличии;
- sanitizers: ASan, UBSan, MSan/TSan по сценарию;
- SAST/SCA/DAST/fuzzing как gates CI/CD;
- traceability: требование -> реализация -> тест -> дефект/уязвимость -> исправление.

## 6. SCA, SBOM и лицензии

Для лекций `kapo`, `spdx`, `ppk`, `sast`, `tz`, `fstec-sdlc`, `new-courses` проверить:

- различение SCA, SBOM, PPK, dependency inventory и license compliance;
- форматы SPDX и CycloneDX;
- источники уязвимостей: БДУ ФСТЭК, NVD, OSV, vendor advisories;
- license compatibility, copyleft, permissive licenses, notice/attribution;
- policy gates: запретные лицензии, unknown license, vulnerable dependency, unsupported component;
- практические evidence: отчет, SBOM-файл, список компонентов, решение по риску.

## 7. Юридическая и этическая безопасность

- Для pentest/offensive-тем есть явные границы законного применения.
- Нет инструкций, которые без контекста разрешения выглядят как призыв к неправомерному доступу.
- Для регуляторики нет категоричных утверждений без ссылки на применимую норму.
- Для сертификации и лицензирования различаются требования закона, методика, практика и экспертное мнение.

## 8. Визуальная и редакторская унификация

- Заголовки короткие и не спорят с карточкой на главной.
- Описания конкретные: чему учим, какой результат, какая практика.
- Цвета, карточки и навигация не мешают чтению.
- Текст не налезает на кнопки, карточки, изображения и соседние блоки.
- Есть единый тон: преподавательский, точный, без лишнего маркетинга.

## 9. Результат аудита

Для каждой лекции фиксировать:

- статус: `ok`, `needs-content-update`, `needs-technical-fix`, `needs-visual-pass`, `needs-editorial-decision`;
- сильные стороны;
- проблемы с приоритетом `P0/P1/P2/P3`;
- рекомендуемые правки;
- какие файлы проверены;
- какие проверки запущены.
